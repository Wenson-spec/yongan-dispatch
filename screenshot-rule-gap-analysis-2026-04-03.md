# 截图规则对照与未落地项整理（2026-04-03）

## 说明

本文依据本轮已核对的页面与代码结果整理，目标是把用户截图中讨论过的业务口径，落成一份可继续执行的 **“已落地 / 未落地 / 待验证”** 对照表。这里的结论主要针对当前系统中用户能直接看到的页面表现、页面文案与链路入口，不等同于对所有后端规则都已完成终验。

当前已重点核对的页面包括 **新建订单、录单台、智能粘贴、找车台、自运派车台、回单管理台、零担统一工作台**。其中，找车台、回单管理台、零担工作台在上一轮已做过一轮明显的前端可见收口；本轮新增发现的缺口，主要集中在 **智能粘贴页、自运派车台，以及零担前后段外请入口是否足够显式** 三类问题。

## 二、已落地项

### 1. 普通外请正式分组优先展示

当前找车台、确认派车摘要、回单跟踪、待交财务、回单管理台已经基本统一到 **“正式外请分组优先、未形成正式分组时回退同车次”** 的口径。用户现在在这些页面上看到的组头、标签、说明条、空态提示，已经不再把旧 `mergedPlanNumber` 当作正式业务组头，而是把它降级成内部参考批次。

这部分已经比较符合截图中“外请相关结果才沉淀成正式分组记录，并且按车、按组展示”的要求，也是当前实际落地最完整的一块。

### 2. 纯零担内部整理结果弱化旧批次号

零担统一工作台待询价区已改为优先显示 **车号、货站、班线、目的站等内部整理执行结果**，而不是把旧批次号直接顶在组头上。空态文案与单条摘要中，也已经明确把参考批次定义为内部对照信息，而非正式外请分组。

这部分已经基本体现了截图里“纯零担的整理结果不做成额外显式正式节点，只体现在零担询价、发运与零担台账里的车号记录”这一方向。

### 3. 新建订单与录单台的待分流语义已基本到位

新建订单页目前明确写着：**订单提交后先进入录单台“待分流”队列，再由录单台分流到外请、自运或零担**。录单台页内也已经形成“待分流 / 退回待处理”的统一处理口径。因此，截图里“录单台承接建议与分流，不直接形成正式外请分组记录”的大方向，已经在这两个页面上得到体现。

## 三、未完全落地项

### 1. 智能粘贴页仍有“已自动分组”旧表述

智能粘贴页当前仍存在类似 **“合并计划号订单已自动分组，点击箭头展开/折叠子单”** 的提示语。这类文案虽然在技术上表达的是前端识别后的分组展示，但在业务语义上很容易被理解成：系统已经替用户完成了正式分组落账。

因此，这一页仍需改成更准确的口径，例如强调其只是 **内部整理 / 参考分组 / 识别归并结果**，而不是正式外请分组记录。

### 2. 自运派车台仍残留旧“合并计划号整组”表达

自运派车台虽然已经补出 **“内部整理参考批次（非正式外请组）”** 的说明，但文件中的顶部说明、分组逻辑注释以及部分组头表达，仍延续了“合并计划号专用”“整组显示”等旧描述。这会削弱截图里“自运可以参与整合/配载判断，但不形成正式整合组记录，只作为系统内部配载结果使用”的边界。

因此，自运派车台还需要继续把页面说明、组头标签和空态提示收口为：**自运内部配载 / 参考批次 / 不形成正式外请分组**。

### 3. 零担前段外请入口尚未完全闭环

现有待办中仍保留了 **“零担待询价阶段前段外请车入口排查”** 的未完成项，说明这部分虽然已经有一些链路修复，但还不能确认页面入口、状态承接和用户提示已经完全落地。按照截图口径，零担前段外请应当是在零担主单下生成前段外请子链，并继续承接定价、审批、派车与送货站动作。

如果页面上没有足够明确的入口、状态标签或链路说明，用户就仍然无法直观看出这条规则已经真正落到系统里。因此，这一项需要继续补查并补做。

### 4. 零担后段外请仍需做一次逐页面显式核对

目前零担后段链路已有目的站、自提送货、结构化收货确认等能力，但是否已经在页面上被明确表达为 **“零担主链完成后，再承接后段外请子链”**，还需要补一轮显式核对。如果只是后端数据结构和状态存在，而页面没有明确告诉用户“当前是后段外请链路”，那仍不能算完全达到截图要求。
