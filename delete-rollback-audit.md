# 退回 / 删除逻辑审计清单

## 审计目标

本轮审计聚焦全系统各工作台、各页签中的**退回**与**删除**按钮，核查其是否满足以下四类一致性要求：

1. **状态回滚正确**：退回后订单、批次、审批、回单等主状态是否回到预期阶段。
2. **关联数据清理完整**：车辆信息、回单记录、押金状态、批次关系、审批记录、子链关系等是否同步清理或保留。
3. **权限与边界一致**：是否存在无权限也能删、已结算仍可退、非当前责任人可操作等问题。
4. **前后端口径统一**：前端按钮显示条件、确认文案、禁用条件是否与后端实际允许条件一致。

## 首轮审计优先级

### P0：先查会直接破坏主数据一致性的动作

- 订单单删 / 批量删除
- 订单单条退回 / 批量退回 / 指定状态退回
- 零担派车批次删除 / 从批次移除订单
- 审批驳回导致的订单状态恢复

### P1：再查会造成页面展示与真实状态不一致的动作

- 回单相关按钮的前后端允许条件
- 模板、辅助数据的软删除口径
- 各工作台按钮显示条件与禁用条件

## 已核实的首轮问题结论

| 模块 | 入口 | 当前实现事实 | 已识别风险 |
| --- | --- | --- | --- |
| 订单 | `delete` | 直接按 `id` 删除订单，仅记录日志；未校验当前状态，未清理回单、审批、批次、子链、押金等关联数据 | **高风险**：任何进入流程中的订单都有机会被直接删除，容易留下脏数据与孤儿记录 |
| 订单 | `batchDelete` | 直接按 `ids` 批量删除订单，仅做日志汇总；未复用单删校验，也未做逐单状态校验与关联清理 | **高风险**：批量删除绕过边界控制，误删范围更大，且单条/批量口径明显不一致 |
| 订单 | `rollbackStatus` | 存在基础状态回退与部分逆向清洗；会清时间字段、车辆信息、押金和 `pending` 回单，但规则主要覆盖订单主表 | **中高风险**：未见对审批链、零担批次关系、子链联动、合并组边界的统一清理，且需要继续核对与批量退回是否完全一致 |
| 订单 | `batchRollback` | 单独维护一套回退映射与清洗逻辑，而不是复用单条退回规则 | **高风险**：单条退回与批量退回存在长期分叉风险，后续极易出现一边修了另一边没修 |
| 订单 | `revertStatus` | 允许退回到白名单中的多个目标状态，并执行比普通退回更强的逆向清洗 | **中高风险**：与 `rollbackStatus` 属于两套规则体系，前端若未区分清楚，会导致“同名退回”实际效果不同 |
| 零担 | `deleteLtlBatch` | 只删除批次与批次-订单关联，不回写订单状态、不恢复待派车上下文、不清理批次生成的派车痕迹 | **高风险**：批次删掉后，订单可能仍停留在错误状态或残留车辆信息，造成“批次没了但订单已派车”的错乱 |
| 零担 | `removeOrderFromLtlBatch` | 只删除批次关联并清空订单车辆信息 | **高风险**：未看到对订单状态、押金、批次派生记录、回单责任等同步恢复，容易出现“已移出批次但不可再次派车/状态未回滚” |
| 审批 | `applyApprovalReject` | 审批驳回主要恢复订单 `status` 与 `approvalDate`；加价驳回会重算费用，但普通审批驳回未见逆向清洗 | **高风险**：若订单在审批前后已生成车辆/回单/押金等派生数据，仅改状态会造成界面回退了但数据未回退 |
| 前端-指挥台 | 删除/退回/驳回按钮 | 当前按钮展示主要依赖权限和页签，不足以反映后端真实可操作边界 | **高风险**：用户能看到并点到很多理论上不该操作的按钮，导致“前端允许点、后端未必安全拦截” |
| 前端-派车台 | 删除/退回/指定状态退回 | 同时存在普通退回与增强退回两套动作，但页面层面的语义区分与后端差异尚不够明确 | **中高风险**：用户难以预期按钮副作用，容易误触发比预期更强的数据清理 |

## 建议采用的统一完善方案

1. 建立一份**退回/删除状态矩阵**：按“对象 × 当前状态 × 允许动作 × 目标状态 × 关联清理 × 日志 × 权限”统一定义，禁止把规则散落在不同页面和 mutation 中。
2. 将订单的 `delete`、`batchDelete`、`rollbackStatus`、`batchRollback`、`revertStatus` 收口为**统一领域规则函数**，前端任何按钮都只能调用这套规则，不再各自维护一份清洗逻辑。
3. 为删除动作增加**硬性后端守卫**：至少校验是否已结算、是否已签收/已回单、是否属于零担批次、是否存在子链、是否存在已处理审批/回单，再决定允许物理删除、软删除或仅允许退回。
4. 为退回动作增加**统一逆向清洗策略**：车辆、司机、押金、审批状态、回单记录、批次关联、子链关系、合并组状态必须有明确的同步规则，且单条与批量必须复用同一实现。
5. 前端按钮改为**显式可操作性判断**：不只是隐藏/展示，还要给出“为什么不能删除/退回”的提示文案，并在确认弹窗中写明将清理哪些数据。
6. 补一组**高风险回归测试矩阵**：单删、批删、单退、批退、审批驳回、删除零担批次、从批次移除订单，分别验证状态回滚、关联清理、非法状态拦截、单批量一致性。

## 下一步核查方法

1. 逐个前端按钮找到其调用的 mutation。
2. 逐个 mutation 反查允许状态、权限、关联数据清洗逻辑。
3. 按“单条操作 vs 批量操作”对比规则是否一致。
4. 按“按钮出现条件 vs 后端实际可执行条件”对比前后端是否一致。
5. 为每类动作补最小回归测试：
   - 正常成功
   - 非法状态拦截
   - 已结算/已完成不可操作
   - 关联记录同步清理/保留正确
   - 批量操作与单条操作口径一致

## 新增核实细节（第二轮）

### 1. 订单删除与退回规则存在明显分叉

| 入口 | 新增核实事实 | 风险判断 |
| --- | --- | --- |
| `order.delete` | 仅查询订单号后直接执行 `delete from orders where id = ?`，没有任何状态白名单、关联数据探测、事务封装或软删除兜底 | **极高风险**：删除动作与业务状态完全脱钩，容易直接打穿审批、回单、零担批次、子链与日志链路 |
| `order.batchDelete` | 与单删一样直接物理删除，仅比单删多一层日志汇总；没有逐单校验，也没有复用单删守卫 | **极高风险**：批量入口放大误删面，且单条/批量不一致风险持续存在 |
| `order.rollbackStatus` | 单条退回采用独立 `ROLLBACK_MAP` 与清洗逻辑；当前映射中**未包含 `shipped`**，意味着该状态单条退回可能直接报不支持 | **高风险**：同一状态在不同入口是否允许退回出现分叉，用户预期不稳定 |
| `order.batchRollback` | 批量退回维护了另一套 `ROLLBACK_MAP`，其中包含 `shipped -> inquiry_confirmed`，与单条退回映射不一致 | **高风险**：相同订单状态下，“单条可否退回/退到哪一步”与“批量可否退回/退到哪一步”可能冲突 |
| `order.revertStatus` | 指定状态退回再次复制了一套阶段清洗逻辑，与 `rollbackStatus`/`batchRollback` 三套并行维护 | **高风险**：任何一次修复都容易只改其中一套，长期必然漂移 |

### 2. 零担批次删除与移除订单缺少完整回滚

| 入口 | 新增核实事实 | 风险判断 |
| --- | --- | --- |
| `order.deleteLtlBatch` | 仅删除 `ltlDispatchBatchOrders` 与 `ltlDispatchBatches` 两张表记录，没有把批次内订单回写到可继续派车的状态，也未清理车辆、司机、押金、回单等派生信息 | **极高风险**：批次删除后订单可能仍停留在“已派车/已绑定车辆”的假状态 |
| `order.removeOrderFromLtlBatch` | 仅删除批次关联，并清空 `plateNumber / driverName / driverPhone`；**没有清空 `driverId / vehicleId`，也没有恢复状态、押金、回单或批次派生记录** | **极高风险**：出现“表面移出批次、底层仍残留派车关系”的半回滚状态 |

### 3. 前端按钮语义与后端真实副作用仍未完全对齐

| 页面 | 新增核实事实 | 风险判断 |
| --- | --- | --- |
| `DispatchVehicle.tsx` | 页面同时提供“退回上一流程”和“退回到指定状态”两套交互，但二者对应后端是不同 mutation、不同规则体系；确认弹窗统一写着会清理派车信息/押金/回单，容易让用户误以为两者效果一致 | **高风险**：前端语义模糊，用户无法准确预估按钮副作用 |
| `DispatchVehicle.tsx` | 删除按钮仅在 `pending` 页签展示，但退回按钮在多个页签都可见；页面展示条件主要依赖页签和权限，没有同步体现订单是否属于零担批次、是否已有审批、是否已有回单操作等真实限制 | **高风险**：前端“能点”不代表后端“应该允许”，容易形成误导性操作入口 |

### 4. 当前最值得优先落地的统一修复原则

1. **先统一后端规则，再收前端按钮。** 删除、单条退回、批量退回、指定状态退回、零担批次删除、从批次移除订单，都应先收口到统一领域规则函数。
2. **把“允许动作判断”与“逆向清洗计划”拆成两个明确步骤。** 先判断当前状态/关联是否允许，再生成需要回滚或清理的字段与关联表操作。
3. **所有高风险动作必须放进事务。** 尤其是订单删除、批量退回、审批驳回、批次删除，必须保证主表状态与关联清理同成同败。
4. **单条与批量必须复用完全相同的规则实现。** 批量只能是“循环执行单条规则并聚合结果”，不能再维护一套独立映射表。
5. **前端改为“可解释的禁用”。** 不只是隐藏按钮，还要明确提示：为什么不能删、为什么不能退、退回后会清掉哪些数据。

### 1. 各工作台当前实际调用口径

| 页面/工作台 | 前端动作 | 当前调用 mutation / 实现 | 新增观察 |
| --- | --- | --- | --- |
| `EntryStation.tsx` 录单台 | 单删 / 批删 | `trpc.order.delete` / `trpc.order.batchDelete` | 录单台的“全部订单/订单总表”也沿用同一删除入口，说明**录单页已不只是草稿态删除**，而是在更大范围暴露了高风险物理删除能力 |
| `CommandCenter.tsx` 指挥台 | 单退 / 批退 / 单删 / 批删 / 审批驳回 | `trpc.order.rollbackStatus` / `trpc.order.batchRollback` / `trpc.order.delete` / `trpc.order.batchDelete` / 审批弹窗 `reject` 流程 | 指挥台同时承载审批驳回与订单退回，前端入口密集，若后端规则不统一，最容易出现**同票多种回退方式效果不一致** |
| `FindVehicle.tsx` 找车台 | 单退 / 指定状态退回 / 批退 / 单删 / 批删 / 审批驳回 | `rollbackStatus` / `revertStatus` / `batchRollback` / `delete` / `batchDelete` / `reject` | 找车台已经同时暴露两套退回 mutation，且页面上都被归类为“退回”语义，用户侧几乎无法直观看出副作用差异 |
| `DispatchVehicle.tsx` 派车台 | 单退 / 指定状态退回 / 批退 / 单删 | `rollbackStatus` / `revertStatus` / `batchRollback` / `delete` | 派车台是车辆、司机、押金、回单等派生数据最集中的位置，但前端仍直接复用订单删除入口，风险最高 |
| `LtlInquiryStation.tsx` 零担找车/询价台 | 单退 / 批退 / 单删 | `rollbackStatus` / `batchRollback` / `delete` | 零担链路页签已能直接删订单，但当前审计尚未看到足够的零担子链/批次保护 |
| `LtlUnifiedWorkspace.tsx` 零担统一工作台 | 单退 / 指定状态退回 / 批退 / 单删 | `rollbackStatus` / `revertStatus` / `batchRollback` / `delete` | 目的站回退场景已经把“结构化备注”与强制退回结合，但底层仍依赖订单层多套退回规则，说明**业务描述在前端增强了，领域规则却未统一** |
| `LtlWorkspace.tsx` 零担工作台 | 单退 / 批退 / 单删 | `rollbackStatus` / `batchRollback` / `delete` | 与链路页共用订单删除/退回接口，存在跨页签同名动作副作用不一致的潜在问题 |
| `LtlDispatch.tsx` 零担派车台 | 删除批次 | `trpc.order.deleteLtlBatch` | 删除批次入口已经前端直连高风险服务端接口，必须优先加后端守卫与事务回滚 |
| `PodDepositStation.tsx` 回单管理台 | 单退 / 批退 / 单删 | `rollbackStatus` / `batchRollback` / `delete` | 回单台中直接调用订单删除/退回，意味着回单阶段的操作实际上会改动订单主流程，前端需要更明确告知影响范围 |
| `OrderPool.tsx` 订单池 | 单退 / 指定状态退回 / 批退 / 单删 / 批删 | `rollbackStatus` / `revertStatus` / `batchRollback` / `delete` / `batchDelete` | 订单池与录单台总表并存，两个“全量视图”都暴露高风险动作，后续需要统一前端可操作性判断 |

### 2. 新增前后端一致性结论

1. **订单层删除接口已经被多个核心工作台复用。** 这意味着 `order.delete` / `order.batchDelete` 不再只是某个孤立页面的问题，而是系统级高风险入口；只要后端没有加硬性状态守卫，多个工作台都会把风险同步放大。
2. **“普通退回”与“指定状态退回”在前端普遍并存。** 尤其在找车台、派车台、零担统一工作台与订单池，页面同时暴露 `rollbackStatus` 与 `revertStatus`，但按钮语义、确认文案和副作用说明尚不足以支撑用户正确判断。
3. **批量动作已经广泛铺开，但仍主要直接调用独立 mutation。** `batchRollback`、`batchDelete` 在多个工作台均为一等入口，因此“批量必须复用单条规则”的治理原则需要尽快落实，否则一处规则分叉会在多个页面持续扩散。
4. **零担与回单工作台并没有形成独立的安全边界。** 零担页签、回单管理台当前仍大量复用订单主流程删除/退回接口，这意味着后端若不按“对象 + 状态 + 关联清理”精细化判断，前端任何一个页签都可能越权触发主流程逆向清洗。

### 3. 对统一修复顺序的进一步建议

建议优先按以下顺序收口：

- 先修 `order.delete` / `order.batchDelete` 的系统级守卫与事务删除计划；
- 再把 `rollbackStatus` / `batchRollback` / `revertStatus` 统一到一套退回规划器；
- 然后单独收 `deleteLtlBatch` / `removeOrderFromLtlBatch`，把零担批次回滚做成专门规则；
- 最后再回到前端，把所有按钮改为基于统一“可操作性判断结果”展示、禁用和提示。
