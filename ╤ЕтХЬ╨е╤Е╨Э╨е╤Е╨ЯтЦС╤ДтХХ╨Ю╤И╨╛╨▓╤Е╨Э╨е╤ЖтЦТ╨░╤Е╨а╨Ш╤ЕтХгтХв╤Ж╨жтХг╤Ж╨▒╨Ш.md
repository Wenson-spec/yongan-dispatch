# 录单台 + 订单池合并后的页面结构与状态流转方案

**项目**：永安调度系统  
**范围**：录单台与订单池合并  
**前提**：**不改变“外请主管指挥台”定位**，仅对录单前台与订单早期处理入口做合并设计。

## 一、方案结论

本方案建议将**“录单台”保留为唯一主入口名称**，而把原来的**“订单池”收编为录单台内部的核心工作视图**。这样做的原因很明确：从业务理解上，录单台代表“订单进入系统的第一站”，而订单池本质上是“这一站里的待处理订单集合”。两者如果继续作为两个并行一级入口存在，用户会在“到底去录单台处理，还是去订单池处理”之间来回切换，造成职责重叠、培训成本上升、退回后承接不清的问题。

合并后，系统对外仍然保留你熟悉的工位名称体系：**录单台、外请主管指挥台、找车台、派车台、零担工作台、回单管理台**。其中，录单台负责“订单进入系统后的首站接收、补录、分流、退回修正、重新指派前处理”；外请主管指挥台仍然只负责外请订单的定价、审核、手动分配与调度前控制，不承担录单入口职责。

| 设计对象 | 合并前问题 | 合并后定位 |
|---|---|---|
| 录单台 | 既像录单入口，又承担部分列表管理 | **主入口工位**，负责接单、补录、分流、退回修正 |
| 订单池 | 与录单台同样能看订单、改订单、筛订单 | **录单台内部视图**，不再单独作为一级导航 |
| 外请主管指挥台 | 容易被上游入口混入早期修单动作 | **保持原定位不变**，只承接外请专业处理 |

## 二、合并后的页面结构

页面结构建议采用“**一个主入口 + 多工作视图**”的方式。也就是说，左侧导航只保留一个一级菜单：**录单台**。原 `/orders` 或“订单池”入口不再对普通用户单独暴露，而是跳转到录单台默认视图，或者仅保留给管理员作为兼容入口。

录单台进入后，页面顶部仍保留“新建订单”“智能粘贴”“批量导入”“刷新”等动作，页面主体则改为多视图工作台。这样既能保留原订单池的筛选、批量、全局查询能力，也能保留录单台的快速录入体验。

| 页面层级 | 名称 | 主要职责 | 是否保留原名称 |
|---|---|---|---|
| 一级导航 | 录单台 | 订单首站入口、早期处理总台 | 是 |
| 一级导航 | 外请主管指挥台 | 外请订单定价、审核、人工分派 | 是 |
| 一级导航 | 找车台 | 外请找车与审批后续 | 是 |
| 一级导航 | 派车台 | 自运派车与执行 | 是 |
| 一级导航 | 零担工作台 | 零担询价、发运与跟踪 | 是 |
| 录单台内部视图 | 订单池 | 默认视图，展示当前待处理与待修正订单 | 是，作为内部标签保留 |
| 录单台内部视图 | 全部订单 | 全量检索、追踪、查看历史 | 新增为内部标签 |
| 录单台内部视图 | 退回待处理 | 承接各工位退回订单，支持快速修正与重新分流 | 新增为内部标签 |
| 录单台内部视图 | 草稿/未完成 | 智能粘贴后未补齐、未提交订单 | 可选 |
| 录单台内部视图 | 导出中心 | TMS 导出与录单侧导出能力 | 保留 |

从交互上看，**“订单池”不消失，但它不再是一个单独系统入口，而是录单台里的默认页签**。这样既保住了现有业务叫法，也解决了双入口冲突。

## 三、录单台内部视图设计

合并后的录单台，不应该只是一个简单表格页，而应该是一个围绕“订单进入系统后如何被正确送到下一工位”的处理中心。页面建议分为“顶部统计区、筛选区、订单列表区、侧边详情/快捷编辑区”四层。

| 区域 | 建议内容 | 设计目的 |
|---|---|---|
| 顶部统计区 | 今日录单、待分流、退回待处理、加急、我创建/我处理 | 让录单员先看积压和风险 |
| 筛选区 | 业务类型、状态、客户、目的地、加急、日期、是否退回、是否缺字段 | 继承订单池的检索能力 |
| 订单列表区 | 支持分组、批量选择、列配置、快捷编辑、批量动作 | 保留订单池的操作效率 |
| 详情侧栏 | 基础信息、收发货信息、业务类型、退回原因、流转记录、快捷提交 | 提升修正与重分流效率 |

在默认的“订单池”视图里，建议把列表重点放在**待进入下一工位的订单**上，而不是把所有后期状态订单混在一起。默认展示的数据范围，应以 `pending_assign` 为主，并额外合并“退回待处理”“草稿未提交”“自动分派失败”等需要录单台继续处理的记录。至于已经进入外请主管指挥台、找车台、派车台后的订单，录单台仍可在“全部订单”里查到，但不应再作为默认工作负载展示。

## 四、状态流转方案

本方案建议把**`pending_assign` 恢复为统一的录单首站状态**。这一步非常关键，因为如果新建订单创建后直接分别落到 `pending_price`、`pending_dispatch`、`pending_inquiry`，那么录单台就很难成为真正统一的上游入口；用户会觉得订单一创建就“飞走了”，而退回时又飞回来，最终入口语义会变乱。

因此，更稳的方案是：**所有新建订单先进入录单台订单池，由录单台确认业务类型、补齐信息并执行分流**。分流完成后，再进入各专业工位。

| 业务类型 | 建议首站状态 | 录单台提交后进入 | 下游工位 |
|---|---|---|---|
| 外请 | `pending_assign` | `pending_price` | 外请主管指挥台 |
| 自运 | `pending_assign` | `pending_dispatch` | 派车台 |
| 零担 | `pending_assign` | `pending_inquiry` | 零担工作台 |

基于这个原则，建议采用如下状态流：

| 流程类型 | 推荐流转 |
|---|---|
| 外请 | `pending_assign` → `pending_price` → `priced` → `pending_vehicle` → `pending_approval` → `dispatched` → `delivered` → `signed` → `settled` |
| 自运 | `pending_assign` → `pending_dispatch` → `dispatched`/`in_transit` → `delivered` → `signed` → `settled` |
| 零担 | `pending_assign` → `pending_inquiry` → `inquiry_confirmed` → `shipped`/`dispatched` → `delivered` → `signed` → `settled` |

这里最重要的不是状态数量是否完全变化，而是**谁负责把订单从 `pending_assign` 正确送到下一站**。这项职责必须完全属于录单台，否则录单台与订单池合并后仍然只是“一个大列表”，而不是“真正的首站工位”。

## 五、退回、重分派与异常闭环

你前面提到的关键问题是对的：订单自动分配后，如果退回了，系统往往只有“退回”动作，没有真正完整的“重新承接、修正、再分流”闭环。这个问题必须在合并方案里一次性补齐。

合并后的设计里，所有来自下游的“需要上游修正”的订单，都统一回到**录单台 > 退回待处理**视图。退回不能只是改状态，还必须附带结构化信息，包括：退回来源工位、退回原因、建议修改字段、是否需要重指派、原负责人、退回时间、最近一次处理人。这样录单员打开页面就知道该怎么修，而不是只看到一个状态值变回去了。

| 场景 | 回退目标 | 回到哪一页处理 | 处理完成后的动作 |
|---|---|---|---|
| 外请定价前信息缺失 | `pending_assign` | 录单台 > 退回待处理 | 补齐后重新提交到外请主管指挥台 |
| 外请区域匹配错误 | `pending_assign` 或保留 `pending_price` | 录单台修正目的地或业务属性 | 重新分流/重新建议指派 |
| 自运派车前基础资料错误 | `pending_assign` | 录单台 > 退回待处理 | 修正后重新提交派车台 |
| 零担询价前信息错误 | `pending_assign` | 录单台 > 退回待处理 | 修正后重新提交零担工作台 |

对于“重新指派”，不建议只在外请主管指挥台或订单池里零散提供入口，而应该在录单台里明确提供两类动作：**重新分流**和**重新指派建议**。重新分流，是把订单改为外请、自运、零担中的另一类；重新指派建议，则是在不改业务类型的前提下，重算推荐工位/推荐调度员/推荐区域。这样退回后订单就能在一个页面里完成闭环，不必来回跳工位。

## 六、权限边界与字段锁定

合并并不意味着录单台能无限改任何订单。恰恰相反，合并后更需要明确权限和步骤锁定，不然它会演变成“谁都能改”的万能页面。

| 角色 | 在录单台可执行动作 | 不应执行的动作 |
|---|---|---|
| 录单员 | 新建、补录、改基础字段、确认业务类型、提交下游、处理退回、批量导入、批量分流 | 不应做外请定价、审批、找车审批 |
| 外请主管 | 查看上游信息、承接 `pending_price`、定价、审核、人工分派 | 不承担新建录单入口 |
| 外请调度员 | 处理 `pending_vehicle`、待审批后续 | 不应修改录单基础字段 |
| 车队调度/现场管理员 | 处理 `pending_dispatch` | 不应承担录单分流职责 |
| 管理员 | 全局查看、紧急接管、回溯审计、配置规则 | 不替代工位日常职责 |

同时，建议继续执行你已经明确要求过的两条底线。第一条是**按步骤锁定字段**：订单进入下一工位后，上一步关键字段默认锁定，只有退回后才允许重新编辑。第二条是**同单并发保护**：同一订单被两个人同时操作时，后提交者必须收到冲突提示，不能静默覆盖。

> 建议把“基础字段锁定”“状态推进锁定”“退回解锁”“版本冲突校验”作为录单台合并方案的基础能力，而不是附属优化项。

## 七、推荐的批量操作能力

你之前强调“合并后功能要保全，尤其要支持批量操作”，这一点我完全同意。因此，订单池原有的批量能力不能丢，而是要迁入录单台内部。

| 批量动作 | 是否建议保留 | 放在哪个视图最合适 |
|---|---|---|
| 批量修改业务类型 | 保留 | 订单池 / 退回待处理 |
| 批量补齐公共字段 | 保留 | 订单池 |
| 批量提交下游 | 保留 | 订单池 |
| 批量退回 | 保留，但仅限有权限角色 | 全部订单 / 管理视图 |
| 批量重新分派建议 | 保留 | 退回待处理 |
| 批量导出 | 保留 | 导出中心 |
| 批量删除 | 谨慎，仅限草稿或管理员 | 草稿/未完成 |

批量操作的目标不是让录单台变复杂，而是保证它在被合并后，仍然能替代原订单池的效率价值。否则表面上是合并，实际上是降级。

## 八、实施方式建议

从落地角度，我建议采用“**先收口入口，再统一状态，再补齐退回闭环**”的顺序，而不是一次性大改所有页面。第一阶段先把菜单入口收口：普通用户只看到录单台，原订单池路由保留但跳到录单台内部视图。第二阶段再统一状态，让新建订单先进入 `pending_assign`。第三阶段补退回待处理、重分流和重指派建议。第四阶段再逐步把订单池原有的列配置、批量视图、保存筛选方案等高级能力完整迁入。

| 实施阶段 | 目标 | 风险控制重点 |
|---|---|---|
| 第一阶段 | 菜单与入口合并 | 不影响旧链接访问，保留兼容跳转 |
| 第二阶段 | 新建订单统一进入 `pending_assign` | 校验报表和积压统计口径同步调整 |
| 第三阶段 | 退回待处理、重新分流、重新指派闭环 | 避免退回后出现无承接工位 |
| 第四阶段 | 迁移订单池高级批量能力 | 保证老用户操作效率不下降 |

## 九、最终建议

如果要一句话概括这版方案，那么就是：**保留“录单台”这个工位名，保留“订单池”这个业务叫法，但让订单池成为录单台里的默认工作视图；同时恢复 `pending_assign` 作为统一首站状态，把录单、修单、退回、重新分流都收回录单台，外请主管指挥台只做外请专业承接，不再承担上游入口职责。**

这样设计以后，系统的名字不变，工位分工更清楚，退回闭环更完整，培训难度也会明显下降。
