## 三、测试过程中发现并修复的问题

在从v3到v4的迭代过程中，共发现并修复了**13个测试失败**，其根因分析如下：

### 3.1 审批流程触发方式错误（6个测试失败）

**问题描述：** v3版本的测试脚本试图通过将订单状态直接改为 `dispatched` 来触发审批，但系统的实际审批流程要求先将状态从 `pending_vehicle` 改为 `pending_approval`，审批通过后系统才会自动将状态变为 `dispatched`。

**根因：** 测试脚本对业务流程的理解有误。审批是在 `pending_vehicle → pending_approval` 这一步触发的，而非在 `dispatched` 阶段。审批通过后，`approval.execute` 会自动将订单状态设为 `dispatched`，并将押金状态设为 `paid`。

**修复方案：** 重写审批测试流程，按照正确的业务链路执行：创建订单 → 定价 → 确保 `pending_vehicle` → 改为 `pending_approval` → 查询审批记录 → 执行审批通过/驳回。

### 3.2 配置管理API返回格式不匹配（4个测试失败）

**问题描述：** v3版本的测试假设 `customer.list`、`warehouse.list`、`freightStation.list` 返回 `{items: [...]}` 格式，但实际这些API返回的是**纯数组**。

**根因：** 配置管理路由的列表接口与订单列表接口的返回格式不同。订单列表返回分页对象 `{items, total, page, pageSize}`，而配置列表返回简单数组。

**修复方案：** 测试中使用兼容性处理：`const list = Array.isArray(res) ? res : (res.items || []);`，同时修正了传参方式。

### 3.3 状态转换测试逻辑错误（2个测试失败）

**问题描述：** v3版本试图通过 `updateStatus` 执行 `pending_assign → pending_price` 的状态转换，但此转换只能通过 `priceAndAssign` 专用接口实现，不在 `updateStatus` 的状态转换矩阵中。

**修复方案：** 重写测试，改为验证 `cancelled` 状态不可恢复的场景。

### 3.4 权限断言错误（1个测试失败）

**问题描述：** v3断言"录单员无权创建客户"，但录单员（`order_entry`）角色确实拥有 `CONFIG_CUSTOMER` 权限——这是业务设计如此，录单员在录入订单时需要能够快速创建新客户。

**修复方案：** 将测试改为正向验证，并新增财务助理无权创建客户的反向验证。

### 3.5 回单操作API路径错误（2个测试失败）

**问题描述：** 测试调用了不存在的 `pod.markSent` 和 `pod.confirmReceived`。

**根因：** 回单状态更新统一通过 `pod.updateStatus` 接口实现，通过 `originalStatus` 参数指定目标状态。

**修复方案：** 改为 `pod.updateStatus({ id, originalStatus: "sent" })` 和 `pod.updateStatus({ id, originalStatus: "received" })`。

## 五、待改进项（非阻塞性问题）

以下问题不影响系统正常运行，但建议在后续版本中改进：

### 5.1 后端金额字段缺少数值格式校验（P2-中）

**影响范围：** `order.create`、`order.updateStatus`、`order.priceAndAssign`、`order.updateOrderFields`

当前所有金额字段（`dispatchPrice`、`actualFreight`、`deliveryFee`、`extraFee`、`customerPrice`、`quotedPrice`等）的zod校验均为 `z.string().optional()`，没有正则校验确保传入的是合法数字格式。理论上可以传入 `"abc"` 或 `"--100"` 等非法值，虽然数据库的 `decimal` 类型会在存储时报错，但错误信息对用户不够友好。

**建议修复：** 添加 `z.string().refine(v => !isNaN(parseFloat(v)) && parseFloat(v) >= 0, "金额必须为非负数字")`。

### 5.2 负数重量未被后端拦截（P3-低）

**影响范围：** `order.create`

`weight` 字段为 `z.string().optional()`，可以传入负数。虽然前端有输入限制，但后端缺少校验。

### 5.3 前端防重复提交（P2-中）

**影响范围：** 所有表单提交按钮

后端已有重复操作拦截（如重复退押金），但前端按钮在提交过程中应禁用并显示加载状态，防止用户快速双击导致重复请求。

### 5.4 录单台表单验证提示优化（P2-中）

空表单提交时验证提示不够明显，建议增加字段红框高亮和错误文字提示。

### 5.5 updateStatus权限细化（P3-低）

当前 `updateStatus` 使用 `protectedProcedure`（仅需登录），建议改为 `permissionProcedure` 以实现更细粒度的权限控制。
