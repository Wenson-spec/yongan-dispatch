# Batch 14 变更说明

## 变更概述

本批次基于 `yongan-batch13-output.zip` 的最新代码，对 `order-flow-review.md` 第九章提出的 **7 条改进建议** 逐项完成修复，并保持了向后兼容原则。所有新增数据字段均以 **optional / nullable** 方式接入，旧数据与旧备注标签逻辑仍可继续识别，避免对存量订单与既有流程造成破坏性影响。

本次修改同时覆盖了后端工作流校验、状态机约束、零担结构化建模以及前端订单池与总览页的公共视图标识展示，并已完成 TypeScript 编译检查。

## 逐项修复说明

### 9.1 拆分“录单待分流”和“退回待处理”

系统已为 `pending_assign` 队列补充显式结构化标识，用于区分新建待分流与退回待处理两种来源。后端在建单、退回与相关流转场景中写入显式队列原因字段，前端展示逻辑可直接读取该标识，无需再依赖日志文本进行人工判断。

这一调整降低了录单台识别成本，并为后续审计报表、责任追踪和统计分析提供了更稳定的数据基础。对于存量历史数据，系统仍保留兼容识别逻辑，避免因为老数据未携带新字段而影响页面显示与流程判断。

| 修复点 | 实施内容 |
| --- | --- |
| 队列来源区分 | 为 `pending_assign` 引入显式来源标识 |
| 写入时机 | 覆盖新建、退回、状态回流等关键场景 |
| 兼容策略 | 保留旧日志 / 旧数据兼容识别 |

### 9.2 统一“分配”接口语义

后端已将“仅设置负责人”和“推进到找车台/对应业务工作台”的语义进行清晰拆分，并在主工作流中加强接口约束，避免形成“负责人已变更但状态未推进”的半流转状态。这样可以显著降低调用方误用不同接口所造成的流程歧义。

前端侧同步朝单一主操作收敛，页面上的工作流展示更强调正式责任工位与当前主状态，减少了用户对“指派”和“流转”两种动作混用的认知负担。

| 修复点 | 实施内容 |
| --- | --- |
| 语义拆分 | 区分仅负责人变更与状态推进型分配 |
| 流程收口 | 主流程优先暴露符合工作流语义的操作 |
| 风险控制 | 减少半流转状态与误调用问题 |

### 9.3 在 `pending_assign` 分流时强校验 `businessType`

后端已将分流映射规则固化为结构化校验，明确要求 `outsource -> pending_price`、`self -> pending_dispatch`、`ltl -> pending_inquiry`。当录单台尝试将订单推进到与业务类型不匹配的目标状态时，系统会直接拒绝该请求。

该改动保留了统一建单入口，但把关键工作流边界收束到后端校验层，确保不同业务线只能进入与自身匹配的处理链路，从而避免人为误推进造成后续责任混乱或数据污染。

| 业务类型 | 允许进入的主状态 |
| --- | --- |
| `outsource` | `pending_price` |
| `self` | `pending_dispatch` |
| `ltl` | `pending_inquiry` |

### 9.4 收紧 `on_hold` 的恢复范围

系统已加强等通知恢复逻辑，不再仅校验“目标状态是否合法”，还会同步校验恢复该状态所需的关键字段是否齐备。对于不同目标状态，恢复条件已进一步细化并纳入统一工作流守卫。

例如，恢复到 `dispatched` 时必须具备可恢复的派车信息；恢复到 `signed` 时必须具备签收依据；恢复到零担中后段时则必须具备零担主单与子链关系的完整结构。这一改动避免了“形式上恢复成功，但业务信息不完整”的伪恢复状态。

| 恢复目标 | 新增校验要求 |
| --- | --- |
| `dispatched` | 必须存在可恢复的派车信息 |
| `signed` | 必须提供签收依据 |
| 零担中后段 | 必须校验主单 / 子链关系完整性 |

### 9.5 将零担前后段外请升级为结构化字段

针对原先通过备注标签识别的“零担前段外请子链”“零担后段外请子链”，本批次已新增结构化字段用于承载关键关系，包括子链阶段、关联主单集合以及分段模式等信息。后端在识别零担链路时优先读取结构化字段，并保留对旧备注标签的回退兼容。

这意味着系统对零担前后段关系的判断不再强依赖备注文本，从而降低备注格式变化带来的识别风险，也为后续统计分析、链路恢复和审计追踪提供了更稳定的数据结构。

| 新增结构化能力 | 说明 |
| --- | --- |
| `subchainStage` | 标识子链属于前段或后段 |
| `relatedParentIds` | 记录关联主单集合 |
| `ltlSegmentMode` | 记录零担分段处理模式 |

### 9.6 把 `settled`、`merged`、`cancelled` 等终态统一纳入状态机描述

本批次已将原先分散在各处的终态限制收束到统一状态机 / 统一校验逻辑中，使 `settled`、`merged`、`cancelled` 等终态的禁止修改、禁止误流转规则不再游离在主流程之外。后端在状态推进、回退和变更校验时会统一参照终态规则集进行判断。

这一调整提高了状态机描述的一致性，也降低了今后新增流程时遗漏某个终态保护逻辑的风险，使系统的工作流治理更加集中、可维护、可审计。

| 终态治理项 | 说明 |
| --- | --- |
| 终态收口 | 将终态统一纳入状态机与校验器 |
| 修改限制 | 统一禁止不合法的终态变更 |
| 维护收益 | 降低规则分散与遗漏风险 |

### 9.7 在订单池和总览页明确标记“公共视图，不代表唯一归属工位”

前端已在 **订单池** 与 **全局概览** 页面新增显式说明文案，明确提示这些列表属于公共视图，不代表订单唯一归属工位。同时，列表标签中已补充展示 **当前主状态**、**主归属工作台** 与 **公共视图展示原因**，帮助用户理解为何同一订单会出现在公共池或跨工作台总览中。

这一调整可以显著降低业务人员在跨列表查看订单时的认知负担，也能减少“同一订单重复出现是否代表重复归属”的误解。页面引导进一步强调，如需继续处理订单，应进入对应正式工作台执行操作。

| 页面 | 新增内容 |
| --- | --- |
| 订单池 | 页面级提示、主状态标签、主归属工作台、公共池展示原因 |
| 全局概览 | 页面级提示、主状态标签、主归属工作台、公共视图展示原因 |

## 兼容性说明

本次新增字段均采用可选或可空方式接入，因此不会强制要求历史订单立刻补齐新字段。对于依赖旧备注标签或旧流程痕迹的数据，系统仍保留兼容解析与回退判断逻辑，以保证历史数据可继续被识别和展示。

如果后续需要进一步推进数据治理，建议在不影响线上使用的前提下，安排增量回填任务，将历史零担子链关系与 `pending_assign` 队列来源逐步补写为结构化字段，以便后续报表与审计完全摆脱文本推断。
